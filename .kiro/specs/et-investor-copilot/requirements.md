# Requirements Document

## Introduction

ET Investor Copilot is a portfolio-aware signal intelligence agent for Indian retail investors, submitted as a Track 6 entry for the ET AI Hackathon 2026 (AI for the Indian Investor). The system monitors NSE/BSE market data, corporate filings, bulk/block deals, and news — then surfaces actionable, source-cited alerts filtered through the user's personal portfolio context. It is built by a solo developer in under 20 hours using a multi-agent LangGraph architecture backed by Google Gemini and OpenAI APIs.

**Disclaimer**: This system is not licensed financial advice. All alerts are for informational purposes only.

---

## Glossary

- **Alert**: A structured, source-cited notification generated by the system about a market signal relevant to the user's portfolio.
- **Audit_Trail**: A step-by-step log of agent reasoning, data sources accessed, and decisions made during signal processing.
- **Bulk_Deal**: A single transaction on NSE/BSE where the traded quantity exceeds 0.5% of the company's equity shares.
- **Block_Deal**: A single transaction of a minimum 5 lakh shares or minimum value of Rs. 5 crore executed through a separate trading window.
- **ContextEnricher**: The agent responsible for cross-referencing signals with portfolio holdings, earnings history, and management commentary.
- **DataFetcher**: The agent responsible for pulling real-time and historical market data, filings, and news.
- **Filing**: A regulatory document submitted to NSE/BSE/SEBI, including bulk deal disclosures, earnings reports, and corporate announcements.
- **Portfolio**: The user's set of stock holdings stored persistently, including ticker symbol, quantity, and average buy price.
- **RSI**: Relative Strength Index — a momentum oscillator measuring speed and magnitude of price changes, ranging 0–100.
- **SignalDetector**: The agent responsible for identifying patterns, anomalies, breakouts, and bulk/block deal events.
- **AlertGenerator**: The agent responsible for producing source-cited, balanced, actionable alerts with an audit trail.
- **Orchestrator**: The LangGraph state machine that coordinates agent handoffs and branching logic.
- **FII**: Foreign Institutional Investor — a registered entity investing in Indian securities markets.
- **52W_High**: The highest price at which a stock has traded in the past 52 weeks.
- **ModelRouter**: The component responsible for selecting the appropriate LLM model for each task based on complexity and cost.
- **BacktestEngine**: The component that computes historical breakout success rates from actual yfinance price history.
- **FilingScanner**: The component that searches NSE/BSE regulatory filings directly and cross-references them against news to detect unreported signals.
- **Unreported_Signal**: A market signal detected from a regulatory filing for which no corresponding news article exists within the past 24 hours.

## Requirements

### Requirement 1: Portfolio Management

**User Story:** As a retail investor, I want to manage my stock portfolio within the app, so that the system can filter all signals and alerts through my actual holdings.

#### Acceptance Criteria

1. THE Portfolio_Manager SHALL allow a user to add a stock holding by specifying NSE/BSE ticker symbol, quantity, and average buy price.
2. THE Portfolio_Manager SHALL allow a user to remove an existing holding from the portfolio.
3. THE Portfolio_Manager SHALL persist portfolio data to Supabase so that holdings survive page refreshes and sessions.
4. WHEN a user adds a holding with an invalid or unrecognised ticker symbol, THE Portfolio_Manager SHALL display an error message and reject the addition.
5. WHEN a user views the portfolio, THE Portfolio_Manager SHALL display each holding with its current market price, day change percentage, and unrealised P&L.
6. THE Portfolio_Manager SHALL support a minimum of 10 simultaneous holdings without degradation.

---

### Requirement 2: Data Fetching Agent

**User Story:** As the system, I want to continuously fetch market data, filings, and news, so that signal detection has fresh, accurate inputs.

#### Acceptance Criteria

1. WHEN the DataFetcher is invoked for a ticker, THE DataFetcher SHALL retrieve current price, volume, 52-week high/low, and RSI(14) using the yfinance library.
2. WHEN the DataFetcher is invoked for bulk/block deal data, THE DataFetcher SHALL query NSE bulk deal disclosures for the current trading day.
3. WHEN the DataFetcher is invoked for news and filings, THE DataFetcher SHALL use the Tavily search API to retrieve the 5 most recent relevant results with source URLs.
4. IF the primary data source (yfinance) is unavailable, THEN THE DataFetcher SHALL fall back to the Google Gemini API with a grounding search and log the fallback event.
5. IF the Tavily API returns zero results, THEN THE DataFetcher SHALL return an empty result set and log the event rather than raising an unhandled exception.
6. THE DataFetcher SHALL complete data retrieval for a single ticker within 10 seconds under normal network conditions.
7. WHEN fetching data, THE DataFetcher SHALL record each data source URL and timestamp in the Audit_Trail.

---

### Requirement 3: Signal Detection Agent

**User Story:** As the system, I want to detect meaningful market signals automatically, so that investors are alerted to events that may require action.

#### Acceptance Criteria

1. WHEN a stock's closing price equals or exceeds its 52W_High and volume exceeds the 20-day average volume by at least 20%, THE SignalDetector SHALL classify the event as a confirmed breakout signal.
2. WHEN a bulk deal is detected where the seller is a promoter or promoter group entity and the deal size exceeds 1% of total equity, THE SignalDetector SHALL classify the event as a promoter-sell bulk deal signal.
3. WHEN RSI(14) for a stock exceeds 70, THE SignalDetector SHALL attach an overbought flag to any concurrent breakout signal for that stock.
4. WHEN FII holding data shows a reduction of more than 1 percentage point quarter-over-quarter for a stock with an active breakout signal, THE SignalDetector SHALL attach a conflicting-FII-reduction flag to that signal.
5. WHEN two or more conflicting flags are attached to a single signal, THE SignalDetector SHALL classify the signal as a conflicted signal requiring balanced presentation.
6. WHEN a macro event (e.g., RBI rate decision, regulatory change) is detected in news, THE SignalDetector SHALL tag the event with the list of portfolio tickers it may affect based on sector mapping.

---

### Requirement 4: Context Enrichment Agent

**User Story:** As the system, I want to enrich signals with portfolio context and historical data, so that alerts are relevant to the specific investor's holdings and risk profile.

#### Acceptance Criteria

1. WHEN the ContextEnricher receives a signal, THE ContextEnricher SHALL check whether the affected ticker is present in the user's Portfolio and attach a portfolio-match flag if so.
2. WHEN a promoter-sell bulk deal signal is received, THE ContextEnricher SHALL retrieve the last 4 quarters of earnings trajectory (EPS trend) and the most recent management commentary for the affected company.
3. WHEN a conflicted signal is received, THE ContextEnricher SHALL retrieve historical breakout success rate for the same stock over the past 2 years (percentage of breakouts followed by >5% gain within 30 days).
4. WHEN a macro event signal is received for a portfolio ticker, THE ContextEnricher SHALL estimate the directional P&L impact on the user's holding using sector sensitivity heuristics and current holding value.
5. WHEN two simultaneous macro events affect the portfolio, THE ContextEnricher SHALL rank them by estimated absolute P&L impact and attach a priority rank to each.
6. THE ContextEnricher SHALL complete enrichment within 15 seconds per signal.

---

### Requirement 5: Alert Generation Agent

**User Story:** As a retail investor, I want to receive clear, source-cited, actionable alerts, so that I can make informed decisions without needing to research everything myself.

#### Acceptance Criteria

1. WHEN the AlertGenerator receives an enriched signal, THE AlertGenerator SHALL produce an alert containing: signal type, affected ticker, summary, recommended action, confidence level, and at least one cited source with URL.
2. WHEN generating an alert for a promoter-sell bulk deal, THE AlertGenerator SHALL include the filing reference number or URL, deal size as percentage of equity, discount to market price, and a risk assessment distinguishing distress vs routine sale.
3. WHEN generating an alert for a conflicted breakout signal, THE AlertGenerator SHALL explicitly list all conflicting indicators (e.g., RSI value, FII change), present a balanced recommendation that is NOT a binary buy/sell, and cite the data sources for each indicator.
4. WHEN generating an alert for a macro event, THE AlertGenerator SHALL include the estimated P&L impact on the user's holding, the priority rank relative to other simultaneous events, and the reasoning for the prioritisation.
5. THE AlertGenerator SHALL append a disclaimer to every alert stating that the content is not licensed financial advice.
6. WHEN the AlertGenerator uses the LLM to generate alert text, THE AlertGenerator SHALL use Google Gemini as the primary model and fall back to OpenAI GPT-4o if Gemini returns an error or empty response.
7. THE AlertGenerator SHALL record the full LLM prompt, model used, and response in the Audit_Trail.

---

### Requirement 6: Multi-Agent Orchestration

**User Story:** As the system, I want a reliable orchestration layer that coordinates agent handoffs and handles errors, so that the pipeline completes reliably even when individual steps fail.

#### Acceptance Criteria

1. THE Orchestrator SHALL implement the agent pipeline as a LangGraph state machine with at least 5 sequential steps: DataFetch → SignalDetect → ContextEnrich → AlertGenerate → AuditLog.
2. WHEN a signal is classified as conflicted, THE Orchestrator SHALL branch to an extended enrichment path that retrieves additional historical context before alert generation.
3. IF any agent step raises an unhandled exception, THEN THE Orchestrator SHALL catch the error, log it to the Audit_Trail, and return a partial result rather than crashing the pipeline.
4. THE Orchestrator SHALL pass a shared state object between agents containing: ticker, raw data, detected signals, enriched context, and generated alert.
5. WHEN the pipeline completes, THE Orchestrator SHALL write the final state (including Audit_Trail) to Supabase.

---

### Requirement 7: Audit Trail and Transparency

**User Story:** As a hackathon judge or power user, I want to see the agent's step-by-step reasoning and data sources, so that I can verify the quality and trustworthiness of each alert.

#### Acceptance Criteria

1. THE Audit_Trail SHALL record each agent step with: agent name, action taken, data sources accessed (with URLs), timestamp, and output summary.
2. WHEN an alert is displayed in the UI, THE System SHALL provide an expandable "Show Reasoning" panel that renders the full Audit_Trail for that alert.
3. THE Audit_Trail SHALL record which LLM model was used for each generation step and whether a fallback occurred.
4. THE Audit_Trail SHALL be stored in Supabase and retrievable by alert ID.
5. WHEN a fallback from Gemini to GPT-4o occurs, THE Audit_Trail SHALL record the reason for fallback (error message or empty response).

---

### Requirement 8: Frontend Dashboard

**User Story:** As a retail investor, I want a clean, responsive dashboard, so that I can view my portfolio, monitor alerts, and inspect agent reasoning in one place.

#### Acceptance Criteria

1. THE Dashboard SHALL display the user's portfolio holdings with live price data in a table format.
2. THE Dashboard SHALL display a feed of generated alerts sorted by recency, with signal type badge, affected ticker, and summary visible without expanding.
3. WHEN a user clicks an alert, THE Dashboard SHALL expand the alert to show full details including recommended action, cited sources, and the "Show Reasoning" audit trail panel.
4. THE Dashboard SHALL include a manual "Run Analysis" button that triggers the full agent pipeline for the current portfolio.
5. WHEN the agent pipeline is running, THE Dashboard SHALL display a progress indicator showing which agent step is currently executing.
6. THE Dashboard SHALL be responsive and usable on both desktop and mobile viewports.
7. THE Dashboard SHALL display a persistent disclaimer banner stating that alerts are not licensed financial advice.

---

### Requirement 9: Source Citation and Evidence Chain

**User Story:** As a retail investor or judge, I want every alert to cite its exact data sources with verifiable references, so that I can trust the alert and trace it back to primary evidence.

#### Acceptance Criteria

1. THE AlertGenerator SHALL attach a structured "Evidence Chain" to every alert, listing each data point used (e.g., RSI value, bulk deal filing URL, EPS figure) with its source name, URL or filing ID, and retrieval timestamp.
2. WHEN an alert references a bulk deal, THE AlertGenerator SHALL include the exact NSE bulk deal disclosure URL or filing reference number — not a generic news link.
3. WHEN an alert references RSI or volume data, THE AlertGenerator SHALL state the exact numeric value and the date it was computed, not a qualitative description.
4. WHEN an alert references management commentary, THE AlertGenerator SHALL cite the specific earnings call transcript or filing from which the commentary was extracted, including the date.
5. THE System SHALL render the Evidence Chain in the UI as a collapsible list of clickable source links beneath each alert.

---

### Requirement 10: Conflicting Signal Resolution Engine

**User Story:** As a retail investor, I want the system to explicitly surface and explain conflicting signals rather than hiding them, so that I understand the full risk picture before acting.

#### Acceptance Criteria

1. WHEN two or more indicators for the same stock point in opposite directions, THE SignalDetector SHALL produce a Conflict_Report listing each indicator, its direction (bullish/bearish), and its magnitude.
2. THE AlertGenerator SHALL render the Conflict_Report as a structured "Bull Case vs Bear Case" section within the alert, with each side's evidence cited separately.
3. WHEN generating a recommendation for a conflicted signal, THE AlertGenerator SHALL use a confidence scale (Low / Medium / High) and SHALL NOT produce a binary "Buy" or "Sell" recommendation.
4. THE AlertGenerator SHALL include a "What to Watch" section in conflicted alerts, listing 2–3 specific future data points the investor should monitor to resolve the conflict (e.g., "Watch next quarter EPS release", "Monitor FII holding change in next SEBI disclosure").
5. WHEN a conflicted alert is generated, THE Audit_Trail SHALL record the full conflict resolution reasoning including which LLM prompt was used and the model's chain-of-thought.

---

### Requirement 11: Impact Quantification

**User Story:** As a retail investor, I want the system to show me the estimated financial impact of each signal on my portfolio in rupees, so that I can prioritise which alerts need my attention first.

#### Acceptance Criteria

1. WHEN generating an alert for a portfolio holding, THE ContextEnricher SHALL compute an estimated impact range in INR based on the signal type, holding value, and sector sensitivity coefficient.
2. THE AlertGenerator SHALL display the estimated impact as a range (e.g., "Estimated impact: ₹-12,000 to ₹-18,000") rather than a single point estimate, to communicate uncertainty.
3. WHEN two alerts are generated simultaneously, THE System SHALL sort the alert feed by absolute estimated impact (highest first) so the most material event appears at the top.
4. THE AlertGenerator SHALL include a brief explanation of the impact calculation methodology within the alert (e.g., "Based on 8% sector drawdown sensitivity × ₹1.5L holding value").
5. WHEN a holding is not in the portfolio, THE System SHALL still generate the alert but SHALL mark it as "Not in your portfolio" and omit the INR impact calculation.

### Requirement 12: Scenario Pack Coverage

**User Story:** As a hackathon judge, I want the system to correctly handle all three scenario pack scenarios end-to-end with verifiable outputs, so that I can evaluate the agent's reasoning quality across diverse signal types.

#### Acceptance Criteria

1. WHEN a promoter sells ≥1% stake via bulk deal at a discount ≥3% to market price, THE System SHALL generate a complete alert covering: filing citation with URL, distress vs routine assessment with reasoning, earnings trajectory context (last 4 quarters EPS), and a specific recommended action with confidence level.
2. WHEN a stock breaks its 52W_High on above-average volume AND RSI > 70 AND FII holding has decreased, THE System SHALL generate a conflicted-signal alert that explicitly names each conflicting indicator with its numeric value, quantifies the historical breakout success rate as a percentage, presents a Bull Case vs Bear Case section, and produces a non-binary recommendation with a "What to Watch" list.
3. WHEN two simultaneous news events affect the portfolio, THE System SHALL identify which event has greater estimated P&L impact on the user's specific holdings, rank the alerts by absolute INR impact, and include the impact quantification methodology in the alert body.
4. WHEN any of the three scenario pack scenarios is triggered, THE System SHALL complete the full agent pipeline and display the alert in the UI within 30 seconds.
5. WHEN a scenario pack scenario is run, THE Audit_Trail SHALL be fully populated and visible in the UI's "Show Reasoning" panel, demonstrating at least 5 distinct agent steps.

### Requirement 13: Cost-Efficient Model Routing

**User Story:** As a hackathon judge, I want to see intelligent model routing decisions in the audit trail, so that I can verify the system uses expensive models only when necessary and quantify the cost efficiency.

#### Acceptance Criteria

1. THE ModelRouter SHALL route classification tasks (sector tagging, signal type detection, bulk deal promoter detection) to Gemini Flash and SHALL route final alert generation for conflicted signals to GPT-4o.
2. THE ModelRouter SHALL route standard (non-conflicted) alert generation to Gemini Pro.
3. WHEN any agent step invokes the ModelRouter, THE ModelRouter SHALL log the routing decision to the Audit_Trail including: task type, model selected, and estimated cost saved versus using GPT-4o for that task.
4. THE Audit_Trail UI SHALL display a "Model Routing" section showing which model handled each pipeline step and the cumulative estimated cost saved versus an all-GPT-4o baseline (e.g., "Used Gemini Flash for classification — saved ~$0.02 vs GPT-4o").
5. WHEN the pipeline completes, THE System SHALL include the total estimated cost efficiency summary in the audit trail.

### Requirement 14: Real Historical Backtesting

**User Story:** As a retail investor, I want the system to show me the actual historical success rate of a breakout pattern for a specific stock, so that I can assess whether this signal has been reliable in the past.

#### Acceptance Criteria

1. THE BacktestEngine SHALL compute the historical breakout success rate for a given ticker by scanning actual yfinance price history for the past 2 years.
2. WHEN computing the breakout success rate, THE BacktestEngine SHALL identify all past breakout events where the closing price was greater than or equal to the rolling 52-week high on above-average volume, then check whether the price was more than 5% higher 30 days after each breakout.
3. THE BacktestEngine SHALL return: success_rate_pct (float, 0–100), sample_size (integer count of breakout events found), avg_gain_pct (average percentage gain on successful breakouts), and avg_loss_pct (average percentage loss on failed breakouts).
4. IF sample_size is less than 1, THEN THE BacktestEngine SHALL return a result with success_rate_pct set to null and a note indicating insufficient historical data.
5. THE BacktestEngine SHALL use only actual yfinance historical price data and SHALL NOT return hardcoded or estimated values.

### Requirement 15: Portfolio Memory and Personalized Alerts

**User Story:** As a retail investor, I want every alert about a stock I hold to reference my specific holding context, so that I understand the signal's relevance to my personal investment rather than receiving generic market commentary.

#### Acceptance Criteria

1. WHEN generating an alert for a portfolio holding, THE AlertGenerator SHALL include the user's holding duration in days (derived from holdings.created_at), current unrealised P&L on that holding, and the signal's estimated impact as a percentage of the user's total portfolio value.
2. WHEN generating a personalized alert, THE AlertGenerator SHALL open with a personalized line referencing the holding duration, average buy price, and current position value (e.g., "You've held INFY for 87 days at ₹1,450 avg. Today's breakout signal directly affects your ₹2.1L position.").
3. THE System SHALL derive holding duration from the holdings.created_at timestamp stored in Supabase, which is set automatically on insert via DEFAULT NOW().
4. WHEN computing total portfolio impact percentage, THE ContextEnricher SHALL sum the current market value of all holdings and express the signal's estimated INR impact as a percentage of that total.

---

### Requirement 16: Filing-First Unreported Signal Detection

**User Story:** As a retail investor, I want the system to surface signals buried in regulatory filings that have not yet appeared in mainstream news, so that I can act on information before it becomes widely known.

#### Acceptance Criteria

1. THE FilingScanner SHALL search NSE bulk deal disclosures directly using Tavily with the query `site:nseindia.com/companies-listing/corporate-filings/bulk-deals` to detect bulk deal filings.
2. WHEN a bulk deal or insider trade signal is detected from a filing source, THE FilingScanner SHALL cross-reference by searching for news articles about the same event published within the last 24 hours.
3. WHEN a filing-sourced signal has no corresponding news article found within 24 hours, THE System SHALL mark the alert as an "Unreported Signal" and assign it a higher priority rank than equivalent reported signals.
4. WHEN an alert is marked as an "Unreported Signal", THE AlertCard UI SHALL display a "🔍 Unreported Signal" badge visually distinguishing it from alerts sourced from news.
5. THE FilingScanner SHALL record both the filing source URL and the news cross-reference search result (or absence thereof) in the Audit_Trail.
